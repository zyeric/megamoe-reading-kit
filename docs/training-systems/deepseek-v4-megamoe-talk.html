0-3 min

DeepSeek-V4 MegaMoE: from MoE semantics to GPU execution

A one-hour systems talk: what changes when expert parallel MoE is lowered into one fused forward kernel.

Central thesis: MegaMoE is not just a faster grouped GEMM. It is an execution rewrite that moves EP dispatch, expert compute, activation, remote write-back, and combine protocol into one persistent GPU kernel.
What the audience should leave withA lowering map from MoE math to distributed EP to a Blackwell-style fused kernel.
What this is notNot a benchmark talk, not a full CUDA tutorial, and not a claim about unpublished backward kernels.

10-15 min

Why MegaMoE is worth reading

It is a compact example of where LLM training / serving systems are going: closer to GPU hardware and workload-specific execution.

Workload pressureMoE creates many small per-expert GEMMs under imbalance; that hurts utilization if dispatch, GEMM, activation, and combine remain separate phases.
System pressureServing, RL rollout, and high-speed agent workloads care about latency-sensitive long-tail batches, not only large balanced throughput.
Research valueThe reusable idea is the lowering pattern: make ownership, memory movement, and compute scheduling part of one GPU-side protocol.
The right abstraction is not 'one more kernel optimization'. It is 'where should the EP boundary live when the GPU is the first-class design object?'

MoE semantics -> distributed EP -> MegaMoE

The math stays stable; the execution representation changes.

MoE semanticstop-k route -> expert FFN -> weighted sum
->
Distributed EPdispatch routes to expert-owner ranks, then combine back
->
MegaMoE rewritefuse dispatch, compute, activation, write-back, combine protocol
Algorithm layerFor each token: route to top-k experts, run expert FFN, multiply by route weights, sum outputs.
Distributed EP referenceMake rank ownership explicit: token-owner rank dispatches routes, expert-owner rank computes, token-owner rank combines.
MegaMoE rewriteKeep the same semantics, but replace materialized phase boundaries with ring buffers, source metadata, counters, and peer writes inside one kernel.
naive MoE:
  y[t] = sum_i topk_weight[t, i] * expert[topk_idx[t, i]](x[t])

distributed EP reference:
  dispatch routes -> local expert compute -> combine routes back

MegaMoE public forward path:
  pull inputs -> L1 -> activation -> L2 -> push outputs -> local combine

Symmetric memory: what it gives and what it does not

Peer-addressable buffers make remote rank memory usable from the fused GPU kernel, but correctness still comes from the kernel protocol.

What setup givesSame-layout rank-local buffers plus rendezvous produce a peer pointer table that the kernel can map by rank and offset.
Dispatch directionThe expert rank pulls source-token payloads from source ranks into local L1 ring slots.
Combine directionThe expert rank pushes Linear2 route outputs into the token-owner rank's combine_token_buffer slot.
What it is notNot an RPC system by itself, not proof of cross-node RDMA support, and not a substitute for counters, barriers, and memory ordering.

37-43 min

Scheduling units that should not collapse together

Wave, ring block, GEMM tile, and top-k slot live at different layers.

wavehigh-level expert-work window
pool blocklogical expert-token range
ring blockreusable producer-consumer slot generation
GEMM tileM x N compute tile with its own K-stage pipeline
top-k slotfinal route-output combine index
WaveA high-level scheduling window that groups a subset of expert work. It helps avoid tiny expert batches starving SMs.
Ring blockA reusable producer-consumer slot generation. It is the unit that counters and spin-waits usually protect.
GEMM tileThe M x N compute tile that UMMA accumulates, with its own K-stage shared-memory pipeline.
Top-k slotThe route-output index used by final combine. It is not necessarily sorted expert-id order.
If a source variable says 'block', first ask: scheduling block, ring block, CTA, GEMM tile, or source-token block?

49-54 min

GEMM and activation detail

Linear1 and Linear2 share the tiled GEMM body, but their epilogues publish different downstream state.

Linear1 bodyTMA loads A/B payloads and SFA/SFB scale factors into shared memory; UTCCP moves scale metadata to TMEM; UMMA accumulates FP32 C in TMEM.
Linear1 epilogueRead gate/up accumulator values from TMEM, round to BF16, apply clamp/SwiGLU and top-k weight, compute amax, write FP8 l2_acts + l2_acts_sf.
Linear2 bodyConsumes l2_acts/l2_acts_sf and L2 weights through the same block-scaled UMMA pattern.
Linear2 epilogueCast route output to BF16 and write it to remote combine slots using TokenSrcMetadata.

54-57 min

Numerics and scale-factor path

The public FP8/FP4 path has a clear dtype boundary around the fused kernel.

before kernelBF16 hidden -> FP8 x + packed x_sf; weights already quantized
Linear1block-scaled UMMA accumulates FP32 C tile in TMEM
activationBF16-rounded gate/up, SwiGLU, top-k weight, FP8 requant
Linear2FP8 l2_acts + weights -> FP32 TMEM accumulator
combineBF16 route outputs, FP32 local slot sum, BF16 y
Before kernelModel-level BF16 hidden states are quantized before launch into FP8 payload plus packed UE8M0 scale factors.
Scale granularityThe main mental model is block-scaled GEMM: payload values are interpreted together with scale-factor metadata, not standalone FP8 arrays.
Activation outputLinear1 epilogue produces FP8 l2_acts/l2_acts_sf for Linear2, so activation is both math and re-quantization.
Final outputLinear2 route output is BF16 in the combine buffer; final combine accumulates slots in FP32 registers and writes BF16 y.

57-60 min

Combine correctness and closing questions

The forward path avoids ambiguous reductions by choosing stable ownership and slot boundaries.

Unique route-output slotEach route writes one BF16 vector to (token-owner rank, token index, top-k slot). No atomic add is needed during write-back.
Barrier before final reduceA phase-level NVLink barrier makes remote combine_token_buffer writes visible before token-owner ranks read local slots.
Fixed local reductionFinal combine walks valid top-k slots in a fixed slot order, accumulates in FP32 registers, and stores BF16 y.
Next questionsBackward dW/dX, cross-node EP, load imbalance policy, and Blackwell/Rubin portability remain the research frontier.
